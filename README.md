# SpyderCheckr Screen Calibrator

**Camera-based monitor calibration using a Datacolor SpyderCheckr 24 and a DSLR/mirrorless camera.**

> ⚠️ **This does not replace a hardware colorimeter.** A dedicated probe (e.g. Spyder X, i1Display) will always be more accurate. This tool is a best-effort calibration for users who already own a SpyderCheckr 24 and a compatible camera, and want better colour than a completely uncalibrated screen.

> This codebase was generated by an AI (Claude, Anthropic). It has not been exhaustively tested on real hardware. Contributions, bug reports, and corrections from human testers are especially welcome.

## How it works
This codebase was generated by an AI (Claude, Anthropic). It has not been exhaustively tested on real hardware. Contributions, bug reports, and corrections from human testers are especially welcome.

```
┌─────────────────────────┬─────────────────────────┐
│   Physical SpyderCheckr │  On-screen target       │
│   (placed in front of   │  (rendered by this app) │
│    the camera)          │                         │
└─────────────────────────┴─────────────────────────┘
                 ↑
         Canon M50 MK2 (RAW, fixed exposure)
```

1. The app splits the screen in two:
   - **Left half**: you place the physical SpyderCheckr 24 card.
   - **Right half**: the app renders the same 24 colour + 8 grayscale patches.
2. The camera shoots both halves in RAW simultaneously.
3. OpenCV detects both patch grids and samples the colour of each patch.
4. A **3×3 correction matrix** is computed (least-squares fit) that maps the *measured screen colours* toward the *SpyderCheckr reference Lab values*.
5. The screen target is updated with the corrected colours and the process repeats for N passes.
6. An **ICC display profile** (`.icc`) is generated and can be applied via `colord` or `xcalib`.

## Requirements

### Hardware
- Datacolor **SpyderCheckr 24** (or SpyderCheckr Classic 24 — same patch layout)
- **Canon EOS M50 MK2** (or any gPhoto2-compatible camera)
- A tripod or stable surface for the camera
- USB cable (camera → PC)

### Software (Linux)
- Python 3.10+
- `gphoto2` CLI + `libgphoto2`
- `dcraw` or `rawtherapee-cli` (RAW conversion)
- `colord` + `xcalib` (optional — for applying the profile)
- Python packages: `opencv-python`, `numpy`, `gphoto2`

## Usage

### GUI mode (recommended)

```bash
python run_calibration.py
```

A control panel appears on the **left half** of the screen.  
The calibration target appears on the **right half** automatically.

**Step-by-step:**
1. Place the physical SpyderCheckr 24 on the **left side of the camera frame**, parallel to the screen.
2. Position the camera so **both the card and the right half of the screen** are visible.
3. Set the camera to **Manual** mode with fixed ISO / shutter / aperture (the app sets this automatically).
4. Click **▶ Start Calibration**.
5. Wait for all passes to complete.
6. Click **🎨 Apply Profile**.

### Headless / scripted mode

```bash
# From a pre-captured photo (no camera needed):
python run_calibration.py --no-gui --image my_photo.jpg

# With live camera capture:
python run_calibration.py --no-gui --passes 1

# Custom output directory:
python run_calibration.py --output ~/calibration_results
```

### CLI options

| Option         | Default    | Description                                 |
| -------------- | ---------- | ------------------------------------------- |
| `--passes N`   | 3          | Number of capture+correction iterations     |
| `--output DIR` | `./output` | Save directory for captures and ICC profile |
| `--no-gui`     | off        | Run headless (no Tkinter window)            |
| `--image PATH` | —          | Headless: use a pre-captured photo          |
| `--debug`      | off        | Enable verbose DEBUG logging                |

## Applying the ICC profile

### Via colord (GNOME / KDE — recommended)

```bash
colormgr import-profile output/calibration.icc
```

Then open **System Settings → Color** and assign the profile to your display.

### Via xcalib (loads VCGT only — limited correction)

```bash
xcalib -d :0 output/calibration.icc
```

> **Note:** xcalib only loads the VCGT (video card gamma table) from the profile, which applies gamma/white-balance correction via the graphics card. The full matrix correction requires colord or a CMS-aware application (GIMP, Darktable, browsers with color management enabled).

## Camera setup tips

For best results:

| Setting       | Recommended value      |
| ------------- | ---------------------- |
| Mode          | Manual (M)             |
| ISO           | 100                    |
| Shutter speed | 1/30 s                 |
| Aperture      | f/5.6                  |
| White balance | Daylight (5500 K)      |
| Picture style | Neutral                |
| Image format  | RAW (CR3)              |
| Stabilisation | Off (camera on tripod) |

- **Shoot in a darkened room** — ambient light on the screen or card will skew measurements.
- **Lens focal length**: 50–85 mm equivalent avoids distortion.
- **Distance**: camera should be far enough that the entire screen half + card are in frame, yet close enough for the card patches to be clearly sampled.

## Understanding ΔE values

The app reports **ΔE76** (CIE 1976 Euclidean distance in Lab space):

| ΔE   | Perception                    |
| ---- | ----------------------------- |
| < 1  | Imperceptible                 |
| 1–2  | Barely perceptible by experts |
| 2–5  | Noticeable to trained eye     |
| 5–10 | Clearly visible               |
| > 10 | Very large difference         |

A typical uncalibrated consumer monitor starts at ΔE ~8–15.  
After 3 passes with this tool, expect ΔE ~3–6 — a meaningful improvement, though not as good as a hardware probe (ΔE < 2).

## Limitations

- **Camera response**: the Canon M50 MK2 has its own spectral sensitivity, which differs from a calibrated colorimeter. Even in RAW, the colours measured through the lens will have some residual error.
- **Lighting**: any ambient light falling on the screen or the physical card will bias measurements. Best results in a dark room.
- **Metamerism**: the camera sensor may confuse colours that look identical to the human eye but reflect differently in near-infrared. This is usually minor with high-quality colour checkers.
- **ICC profile type**: the generated profile is a matrix/TRC type (no 3D LUT). It corrects linear colour shifts well but cannot correct non-linear spatial variations across the screen.

## Contributing

Pull requests welcome. Areas where contributions would be most valuable:

- **Better RAW pipeline**: using `rawpy` (libraw bindings) instead of `dcraw` for more consistent linearisation.
- **ΔE2000 metric**: replacing the ΔE76 with the more perceptually uniform CIEDE2000.
- **3D LUT output**: generating a `.cube` LUT for use in Darktable / GIMP instead of (or in addition to) the ICC profile.
- **Windows support**: testing and adapting the camera/USB path for Windows (`gphoto2` has a Windows build).
- **More colour checkers**: adding reference values for X-Rite ColorChecker Classic 24 and 30-patch variants.
